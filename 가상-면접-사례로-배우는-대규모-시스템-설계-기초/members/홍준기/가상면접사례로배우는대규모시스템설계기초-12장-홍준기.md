# [스터디 자료] 12장: 채팅 시스템 설계 - 핵심 정리

## 1. HTTP와 WebSocket의 관계

### 연결 수립 과정 (Handshake)
웹소켓은 처음 연결을 맺을 때 HTTP 프로토콜을 사용합니다. 이를 '프로토콜 업그레이드'라고 부릅니다.

* **의문점: 왜 처음부터 TCP 소켓을 직접 열지 않는가?**
    * **호환성 문제**: 웹 브라우저 환경에서는 보안상의 이유로 개발자가 임의의 TCP 소켓을 직접 여는 것을 제한합니다.
    * **포트 최적화**: 대부분의 네트워크 장비는 80(HTTP)과 443(HTTPS) 포트에 최적화되어 있어, 해당 포트로 HTTP 접근을 해야 차단 없이 문제없는 첫 연결이 가능합니다.

### 통신 방식의 차이
| 구분 | HTTP | WebSocket |
| :--- | :--- | :--- |
| **연결 유지** | 응답 후 즉시 끊음 (무상태) | 연결 수립 후 계속 유지 (양방향 상태 유지) |
| **메시지 무게** | 무거움 (요청마다 수많은 헤더 포함) | 가벼움 (연결 이후에는 헤더를 포함하지 않음) |

## 2. 웹소켓 기반 프로젝트의 커넥션 관리

### 수평적 확장(Scale-out)의 어려움
HTTP 기반 서비스는 요청이 어느 서버로 가든 상관없는 무상태 설계가 가능합니다. 하지만 웹소켓 환경에서 1,000명의 동시 접속자는 곧 물리적인 1,000개의 커넥션 점유를 의미합니다.

* **상태 공유의 필요성**: 인간1이 서버 B에 접속해 있고, 인간2가 서버 A에 접속해 있을 때, 서버 A는 인간2에게 인간1의 메세지를 보여줄 수 있어야 합니다.
* **해결 방향**: 웹소켓 서버들끼리 상태를 공유하기 위해, 스케일 아웃 관계의 서버들 사이에 메세지 큐(Message Queue)를 두는 것이 일반적인 설계 방식입니다.

## 3. 상태 서버와 박동(Heartbeat) 메커니즘

상태 서버와 박동 메커니즘을 통해 클라이언트의 활성/비활성 상태를 파악할 수 있습니다.
* **커넥션 관리**: 상대방의 비활성화를 감지할 수 있으며, 박동이 끊겼을 때 웹소켓 연결도 종료하여 효율적인 커넥션 관리가 가능합니다.

## 4. 메시지 동기화 및 오프라인 메세지 처리

### 메세지 수신 시나리오 (오프라인 상태)
사용자가 비활성화 상태일 때 누적된 메세지를 다시 접속했을 때 어떻게 짧은 레이턴시로 받아볼 수 있는지에 대한 의문과 해결책입니다.

* **메세지 동기화 로직**: 사용자가 다시 앱을 켜서 웹소켓 연결을 맺는 순간 동기화가 작동합니다.
* **데이터 조회 및 설계 고찰**:
    * 사용자마다 물리적인 큐를 만드는 것은 수천만 개의 큐가 생성되므로 관리와 비용 측면에서 비효율적입니다.
    * **DB 활용**: DB에 `sequence_id`를 부여하여 저장하고, 접속 시 특정 ID에 대해 미수신 테이블에서 현재 접속 시점 이후의 데이터를 한 번에 가져오는(Bulk 조회) 쿼리를 날리는 것이 비용적으로 우수합니다.
    * **비용 절감 구조**: 최근 메세지들은 캐싱을 통해 비용을 절감하며 빠르게 보여주고, 오래된 메세지는 키-값 저장소(DB)에서 가져오는 방식을 취할 것으로 예상됩니다.

## 5. 저장소 선택: NoSQL(키-값 저장소) vs RDBMS

채팅 서버는 일반적인 서버(읽기/쓰기 5:5)와 달리 **쓰기(Write)**가 압도적으로 많습니다.

* **키-값 저장소의 장점**: RDBMS처럼 인덱스가 많지 않아 쓰기 작업에 압도적으로 유리합니다.
* **발생하는 트레이드오프**: 
    * 하나의 Key에 많은 데이터를 담아야 하므로 정규화를 거의 포기해야 합니다.
    * 이로 인해 데이터 중복이 발생하며, RDBMS 입장에서 비효율적으로 보일 만큼 저장해야 할 데이터양이 막대해질 수 있습니다.

**결론적 고찰**: 실시간 채팅 서버는 인프라 비용이 상당히 많이 드는 구조입니다. 채팅 자체로 직접적인 수익을 내는 시스템이 많지 않음을 고려할 때, 기업들이 채팅(메신저)을 기반으로 다른 사업을 문어발식으로 확장하는 이유를 짐작해 볼 수 있습니다.
