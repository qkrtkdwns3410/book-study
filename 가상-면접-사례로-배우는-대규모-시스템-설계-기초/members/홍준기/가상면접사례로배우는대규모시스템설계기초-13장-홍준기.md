# 13장 검색어 자동완성 시스템 설계

## 1. 검색 패러다임의 변화: 검색어 자동완성에서 프롬프트 추천으로
배경: 기존의 단순 검색 시스템에서 대화형 AI로 패러다임이 전환되고 있으며, 대화형 AI에서는 사용자의 '프롬프팅'이 매우 중요해졌다.

관점의 확장: 검색어 자동완성을 대화형 AI 관점에서 보면, 이는 곧 '프롬프팅의 추천'과 같다.

LLM 제공사의 고민: 대화형 AI가 사용자의 프롬프팅 작성까지 지원(추천 및 자동완성)하려면 기술적 한계에 부딪힌다. 사용자가 타자를 칠 때마다 무거운 거대 언어 모델을 호출하는 것은 비용과 처리 속도(Latency) 측면에서 불가능에 가깝다. 따라서 기존처럼 매 타자마다 반응하기보다는, 추천 질문을 칩(Chip) 형태로 띄워주거나 가벼운 캐싱 모델을 앞단에 두는 방식으로 타협점과 최적화를 모색하고 있다.

## 2. RDBMS가 검색어 자동완성에 비효율적인 이유
조회 속도의 함정: "가"라는 검색어에 대해 LIKE '가%'로 찾는 것 자체는 인덱스 스캔을 타기 때문에 매우 빠르다. (RDBMS나 트라이나 찾는 속도는 무의미할 정도로 둘 다 빠름)

정렬 비용의 한계: 문제는 찾은 수많은 "가%" 단어들(가물치, 가위, 가수 등)을 search_count 기준으로 정렬(Order By)하여 상위 10개를 뽑아내는 과정이다.

비효율성: 사용자가 매 타자를 칠 때마다 이 무거운 정렬 작업이 매우 빠르게 수행되어야 하므로 RDBMS는 이 시스템에 적합하지 않다.

## 3. 트라이(Trie) 자료구조의 핵심 원리
구조적 장점: 트라이 자료구조는 노드 탐색 과정에서 정렬을 할 필요가 없다.

Top-10 캐싱: "가"라는 노드 내부에 이미 search_count 상위 10개에 해당하는 단어 목록이 캐싱되어 들어있다.

결과: 따라서 "가"라는 Prefix(접두어) 노드까지만 탐색을 마치면, 추가적인 정렬 연산 없이 상위 10개 결과를 즉시 반환할 수 있다.

## 4. RDBMS 튜닝의 한계: 업데이트 지옥(Update Hell)
질문: RDBMS에도 트라이처럼 Prefix 컬럼을 두고, 10개의 인기 검색어 컬럼을 만들어 데이터를 다 저장해두면 되지 않을까?

문제점: 조회 지옥은 피할 수 있으나, 업데이트 지옥이 열린다. 검색어 자동완성 데이터는 일반적인 경우보다 쓰기(Write) 비율이 월등히 높다.

락(Lock) 이슈: DB에 쓰기 작업이 발생하면 해당 행(Row)에 락이 잡힌다. 특정 행에 100개의 쓰기 요청이 몰리면 99개는 대기해야 하며, 이는 메모리가 아닌 디스크 쓰기이므로 매우 느리다.

대안 및 결론: 굳이 RDBMS를 고집한다면 메시지 큐(Message Queue) 같은 인프라를 완충지대로 두어 쓰기 작업을 모아서 일괄 처리(Batch)해야 한다. 하지만 이 방식 역시 트라이 시스템에 적용하는 것이 훨씬 빠르고 효율적이다.

## 5. 트라이(Trie)와 엘라스틱서치(Elasticsearch)의 비교
엘라스틱서치의 특징: 내부적으로 트라이와 매우 유사한 자료구조를 사용하며, 오타 교정이나 초성 검색 등 고급 기능을 손쉽게 사용할 수 있도록 지원한다.

고트래픽 환경에서의 한계: 구글 메인 화면의 검색창과 같은 초거대 고트래픽 환경에서는 엘라스틱서치조차 무거울 수 있다.

최적화 방향: 이러한 극한의 환경에서는 순수 속도를 끌어올리기 위해 Redis나 C/C++로 가볍게 자체 구현한 '인메모리 트라이/캐시 서버'를 앞단에 배치하여 사용한다.

## 6. 자료구조의 확장: 트라이에서 FST로
질문: 트라이 자료구조에서 자식 노드의 부모는 무조건 하나뿐인가? 부모가 여러 개가 될 수는 없는가?

FST (Finite State Transducer): 엘라스틱서치 등에서 트라이를 더욱 발전시켜 사용하는 자료구조다. 트라이 구조에서 중복되는 접미어(Suffix) 노드들을 하나로 합쳐 메모리 효율을 극대화한다.

구조적 특징: 노드를 합치게 되면서 더 이상 트리(Tree)가 아닌 그래프(Graph) 구조를 띈다.

검색 가능 이유: "그래프라면 루트가 없는데 어떻게 Prefix를 잡고 탐색하는가?"에 대한 답은, FST가 루트(시작점)가 분명히 존재하며 위에서 아래로만 향하는 방향성 비순환 그래프(DAG)이기 때문이다. 따라서 트라이와 동일하게 접두어 탐색이 가능하다.
