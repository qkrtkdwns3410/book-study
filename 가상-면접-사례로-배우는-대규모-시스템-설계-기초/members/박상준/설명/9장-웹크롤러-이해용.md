# 9장 웹 크롤러 - Q&A 정리 (이해용)

---

## Q1. 크롤러는 왜 차단당하나요?

웹사이트들은 **Cloudflare 같은 WAF(웹 방화벽)**를 사용해서 봇/크롤러를 걸러냅니다.
크롤러가 요청을 보내면 실제 서버에 도달하기 전에 **3단계 검문**을 거칩니다.

---

## Q2. 1단계 검문 - IP 평판 검사란?

요청이 들어오면 먼저 **IP의 평판**을 확인합니다. 의심스러운 IP라면 **JavaScript 챌린지**를 내보냅니다.

**JavaScript 챌린지의 핵심 원리:**
- Cloudflare가 JavaScript 코드(예: `a + b` 계산)를 보냄
- **진짜 브라우저**: JS를 실행해서 정답을 쿠키에 담아 다시 요청 → 통과
- **크롤러(Python 등)**: JS를 실행할 수 없어서 정답을 못 구함 → **403 차단**

쉽게 말해, "이 수학 문제 풀어봐"라고 시험을 보는 건데, 크롤러는 시험지를 읽을 줄만 알지 풀 줄은 모르는 셈입니다.

---

## Q3. 2단계 검문 - TLS 핑거프린트란?

**비유**: 사람마다 문을 노크하는 패턴이 다르듯, 브라우저마다 **TLS 연결을 맺는 방식(핑거프린트)**이 다릅니다.

- Chrome은 노크 3번, Python 크롤러는 2번 — 이런 차이를 서버가 감지
- 크롤러가 `User-Agent: Chrome`이라고 거짓말해도, **악수하는 방식(TLS 핸드셰이크)**이 Python이면 들통남

**JA3 핑거프린트** 방식이 가장 널리 쓰입니다:

```
JA3 = MD5(SSL버전 + 암호화스위트 + 확장목록 + 타원곡선 정보)
```

- Chrome의 JA3 해시: `cd08e31fd8d8...`
- Python의 JA3 해시: `51c64c77e60f...` ← 완전히 다름!

저수준 라이브러리에서 결정되는 값이라 **위조가 매우 어렵습니다**.

---

## Q4. 3단계 검문 - HTTP 헤더 분석이란?

진짜 브라우저라면 당연히 있어야 할 헤더들이 있는지 확인합니다:
- `Referer`, `Accept-Language` 등의 **존재 여부**
- 헤더의 **순서**와 **값**이 해당 브라우저와 일치하는지

이 모든 검증을 통과하면 `cf_clearance 쿠키`가 발급되고, 이후 요청은 통과됩니다.
이상 행동이 감지되면 최종적으로 **CAPTCHA**(퍼즐 조립 등)를 띄워서 사람인지 확인합니다.

## Q5. 차단을 우회하는 방법은?

| 방법 | 원리 | 장단점 |
|------|------|--------|
| **IP 로테이션** | 여러 IP를 돌려가며 사용 | 구현 간단, 하지만 비정상 패턴으로 결국 차단될 수 있음 |
| **User-Agent 로테이션** | 브라우저 식별자를 바꿔가며 요청 | 위와 동일하게 단순하지만 한계 있음 |
| **FlareSolverr** | 실제 브라우저(Selenium)로 JS 챌린지를 풀어주는 프록시 서버 | 쿠키 재사용 가능, 하지만 초기 리소스 소모 크고 동시 처리 제한 |
| **ScrapingBee** | 유료 클라우드 API로 차단 우회를 대행 | 직접 구현 불필요, 쿠팡 1,000건 크롤링 시 약 65원 |

---

## Q6. FlareSolverr는 어떤 흐름으로 동작하나요?

```
크롤러 → FlareSolverr에 URL 요청
         → FlareSolverr가 실제 브라우저(Selenium) 실행
           → 브라우저가 Cloudflare JS 챌린지를 풀어냄
             → 통과된 쿠키 + HTML을 크롤러에 반환
```

핵심은 **한 번 챌린지를 풀면 쿠키를 재사용**할 수 있다는 점입니다.
다만 처음 풀 때 리소스가 많이 들고 동시 처리에 한계가 있습니다.

---

## 한 줄 요약

> 웹 크롤러는 IP 평판 → TLS 핑거프린트 → HTTP 헤더의 3단계 검증에 의해 차단되며,
> 우회 방법으로는 IP/UA 로테이션, FlareSolverr(프록시), ScrapingBee(유료 API) 등이 있지만 각각 한계가 존재한다.
