# 10장 알림 시스템

#  Android 푸시 알림

핵심 질문

-   안드로이드 푸시 알림은 **어떤 프로토콜로 전달되는가**
-   내부 **동작 원리는 무엇인가**

이 부분을 조사하기 위해 **FCM(Firebase Cloud Messaging)**을 확인.

------------------------------------------------------------------------

# FCM (Firebase Cloud Messaging)

FCM은

> 서버에서 모바일 앱(Android, iOS, Web)에 **푸시 메시지를 전달하는
> 서비스**

------------------------------------------------------------------------

# 5. FCM 전체 흐름

    [내 서버]
        │
        │ 1. 메시지 전송 요청 (HTTP API)
        ▼
    [FCM 서버 (Google)]
        │
        │ 2. 대상 기기에 메시지 전달 (토큰 기반)
        ▼
    [사용자 스마트폰]
        │
        │ 3. 사용자 스마트폰내의 Google Play Service가 메시지 수신. 앱에 메시지 전달(IPC)
        ▼
    [앱이 알림 표시 or 데이터 처리]

------------------------------------------------------------------------

* 참고 자료
  * https://developers.google.com/android/guides/overview

# 핵심 개념

## 1️⃣ Registration Token (기기 토큰)

앱이 처음 실행되면

    앱 → FCM 등록

FCM이 **기기 토큰을 발급**

예

    eQh2dF8K2dP:APA91bH0...

이 토큰은

> "이 메시지를 어느 기기에 보낼지" 식별하는 주소

### 토큰 생성 흐름

    앱 실행
       ↓
    FCM 등록
       ↓
    토큰 발급
       ↓
    앱 → 내 서버로 토큰 전달

그래서 서버 DB에는 보통

    user_id | fcm_token

형태로 저장된다.

------------------------------------------------------------------------

## 2️⃣ Message

서버가 앱보내는 메시지

예

``` json
{
 "to":"fcm_token",
 "notification": {
   "title":"새 메시지",
   "body":"친구가 메시지를 보냈습니다"
 }
}
```

* 메시지 종류
  * notification message
    * "시스템이 알아서 처리해 주는 편리한 알림"
    * 알림 메시지는 사용자에게 정보를 보여주는 것이 주된 목적입니다. 개발자가 복잡한 코드를 짜지 않아도, 앱이 백그라운드 상태일 때 스마트폰 운영체제(iOS/Android)가 알아서 알림창(Push)을 띄워줍니다.
  * data message
    * "앱이 직접 내용을 열어보고 처리하는 비밀 가방"
    * 사용자에게 바로 알림을 띄우는 대신, 앱 내부적으로 어떤 로직을 처리하고 싶을 때 사용합니다. 알림창을 띄울지 말지, 띄운다면 어떤 디자인으로 띄울지 등을 개발자가 코드로 직접 제어합니다.
* 참고 :
  * https://firebase.google.com/docs/cloud-messaging/customize-messages/set-message-type

------------------------------------------------------------------------

## 3️⃣ Topic / Group

토큰 대신 **그룹으로 메시지 전송 가능**

예

    topic : sports
    topic : news

    서버 → FCM → sports 구독자 전원

# 실제 동작 원리

핵심 아이디어

> 기기와 FCM 서버가 **persistent connection**을 유지한다.

### 기기 내부 구조

    FCM SDK
       ↓
    Google Play Service
       ↓
    FCM 서버와 persistent connection 유지

즉

    스마트폰 ↔ FCM 서버

항상 연결 상태 유지

------------------------------------------------------------------------

## 중요한 특징

앱마다 TCP 연결을 따로 유지하는 구조가 아니다.

대신

> 기기 내부의 시스템 컴포넌트가 **공용 연결을 유지**

그리고

    FCM → 기기
          ↓
    어떤 앱으로 전달할지 내부 분배

이 방식은

-   배터리 절약
-   네트워크 효율

측면에서 유리하다.

Android에서는

-   Doze (잠자기 모드 : Doze는 사용자가 기기를 사용하지 않고 충전 중이 아닐 때, 시스템 전체의 배터리 소모를 줄이는 기능입니다.)
-   App Standby (앱 대기 모드 : App Standby는 사용자가 오랫동안 사용하지 않은 '특정 앱'만 골라서 제한하는 기능입니다.)

환경에서도 동작하도록 최적화되어 있다.

------------------------------------------------------------------------

# 실제 전달 과정

푸시가 전달되는 단계

1️⃣ 앱이 실행되면 FCM 등록 토큰을 받음\
2️⃣ 앱이 토큰을 **내 서버에 전달**\
3️⃣ 서버가 **FCM HTTP API로 메시지 전송 요청**\
4️⃣ FCM이 **토큰으로 대상 기기 확인**\
5️⃣ 이미 열려있는 **푸시 연결을 통해 기기로 전달**\
6️⃣ 시스템이 알림 표시 또는 앱 콜백 실행

------------------------------------------------------------------------

# 9. 앱이 꺼져 있어도 알림이 오는 이유

핵심 이유

> 앱 프로세스가 아니라 **기기 푸시 채널이 살아있기 때문**

즉

    앱 프로세스 종료
            ↓
    기기 OS 푸시 채널(goole Play Service)은 유지

메시지가 도착하면

-   notification message → 시스템이 직접 알림 표시
-   data message → 상황에 따라 앱을 깨워 처리

단

-   배터리 정책
-   OS 상태
-   우선순위

에 따라 즉시성은 달라질 수 있다.
