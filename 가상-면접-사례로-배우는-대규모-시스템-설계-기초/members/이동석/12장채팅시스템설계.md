# 📌 채팅 시스템 설계 핵심 정리

## 1. 요구사항 가정

* DAU: 5,000만

* 사용자당 API 호출: 10회/일
  → **총 5억 요청/일**

* 초당 처리량:

  ```
  500,000,000 / 86,400 ≈ 5,787 RPS
  ```

---

## 2. 메시지 전달 방식 비교

### ✅ Polling

* “지금 메시지 있어?” 계속 요청
* 단점: 불필요한 요청 많음

---

### ✅ Long Polling

* 요청 후 서버가 **응답을 붙잡고 대기**
* 이벤트 발생 시 응답

💡 핵심

> 타임아웃이 아니라
> “서버가 요청을 유지하느냐”가 본질

---

### ✅ WebSocket

* 클라이언트 ↔ 서버 **양방향 연결 유지**
* 서버 → 클라이언트 push 가능

👉 채팅 시스템의 핵심 기술

---

## 3. WebSocket 끊김 & 재연결

### ✔ 프로토콜 수준

* **Graceful Close**

  1. Close frame 전송
  2. 상대가 응답
  3. TCP 종료

---

### ❗ 실제 중요한 것 (구현 영역)

* 끊김 감지 (heartbeat / ping-pong)
* timeout 기준
* 재연결 전략 (retry, backoff)

👉 전부 **서비스 정책에 의존**

---

## 4. 서버 역할 분리

### 4.1 Chat Server

👉 “메시지 처리”

* 메시지 송수신
* 채팅방 fan-out
* 메시지 저장
* 순서 보장

---

### 4.2 Presence Server

👉 “사용자 상태 관리”

* online/offline
* 어느 서버에 연결됐는지 tracking
* last seen / typing
* WebSocket 세션 관리

---

## 5. 메시지 흐름 (End-to-End)

### 📤 1. 사용자 A → 메시지 전송

1. A → Chat Server (WebSocket)
2. Chat Server → 메시지 저장
3. Chat Server → Presence Server 조회

   * “수신자들이 어디 붙어있지?”

---

### 📡 2. 메시지 전달

4. Chat Server → 각 수신자의 연결 서버로 전달
5. 해당 서버 → WebSocket으로 push

---

### 📥 3. 수신

6. 사용자 B → 메시지 수신
7. 필요 시 read 처리 업데이트

---

## 6. Fan-out 전략

### ✅ Fan-out Write (Push)

* 메시지 생성 시 **모든 수신자에게 즉시 전달**

**장점**

* 빠른 전달

**단점**

* 그룹이 크면 비용 폭증

---

### ✅ Fan-out Read (Pull)

* 메시지 1번 저장
* 사용자가 접속 시 가져감

**장점**

* 쓰기 비용 낮음

**단점**

* 실시간성 약함

---

### ✅ Hybrid (실무 표준)

* 온라인 유저 → push
* 오프라인 유저 → pull

👉 대부분의 서비스가 이 구조

---

## 7. 대규모 그룹 채팅 전략

* 소규모:

  * fan-out write 가능

* 대규모:

  * push 비용 폭증
  * 일부는 pull 방식 사용

👉 그래서:

* 카카오톡 등은 **그룹 인원 제한 존재**

---

## 8. Service Discovery

👉 클라이언트가 접속할 서버를 알려주는 역할

* DNS / LB / Discovery 서버
* “어디 서버로 붙어라” 안내

---

## 9. 롱테일(Long Tail) 개념

### ✔ 정의

> 개별은 작지만, 전체 합은 큰 데이터

---

### ✔ 채팅 시스템에서의 예

* 인기 채팅방 (Head)
* 수많은 소규모 채팅방 (Tail)

👉 대부분 트래픽은 Tail에서도 발생

---

### ✔ RDB의 한계

* 고정 스키마
* 다양한 데이터 구조 대응 어려움

---

### ✔ 해결 방식

* NoSQL
* JSON 컬럼
* Key-Value 구조
