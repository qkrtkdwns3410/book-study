## 2. 메시지 전달 방식 비교

### ✅ Polling

* “지금 메시지 있어?” 계속 요청
* 단점: 불필요한 요청 많음

---

### ✅ Long Polling

* 요청 후 서버가 **응답을 붙잡고 대기**
* 이벤트 발생 시 응답

💡 핵심

> 타임아웃이 아니라
> “서버가 요청을 유지하느냐”가 본질

---

### ✅ WebSocket

* 클라이언트 ↔ 서버 **양방향 연결 유지**
* 서버 → 클라이언트 push 가능

👉 채팅 시스템의 핵심 기술

## 4. 서버 역할 분리

### 4.1 Chat Server

👉 “메시지 처리”

* 메시지 송수신
* 채팅방 fan-out(여러 사용자에게 전달하는 방식)
* 메시지 저장
* 순서 보장

---

### 4.2 Presence Server(접속서버)

👉 “사용자 상태 관리”

* online/offline
* 어느 서버에 연결됐는지 tracking
* last seen / typing
* WebSocket 세션 관리

---

## 5. 메시지 흐름 (End-to-End)

### 📤 1. 사용자 A → 메시지 전송

1. A → Chat Server (WebSocket)
2. Chat Server → 메시지 저장
3. Chat Server → Presence Server 조회

   * “수신자들이 어디 붙어있지?”

---

### 📡 2. 메시지 전달

4. Chat Server → 각 수신자의 연결 서버로 전달
5. 해당 서버 → WebSocket으로 push

---

### 📥 3. 수신

6. 사용자 B → 담당 Chat Server로 부터 메시지 수신
7. 필요 시 read 처리 업데이트

---

## 6. Fan-out 전략

### ✅ Fan-out Write (Push)

* 메시지 생성 시 **모든 수신자에게 즉시 전달**

**장점**

* 빠른 전달

**단점**

* 그룹이 크면 비용 폭증

---

### ✅ Fan-out Read (Pull)

* 메시지 1번 저장
* 사용자가 접속 시 가져감

**장점**

* 쓰기 비용 낮음

**단점**

* 실시간성 약함

---

### ✅ Hybrid (실무 표준)

* 온라인 유저 → push
* 오프라인 유저 → pull

👉 대부분의 서비스가 이 구조

---

## 7. 대규모 그룹 채팅 전략

* 소규모:

  * fan-out write 가능

* 대규모:

  * push 비용 폭증
  * 일부는 pull 방식 사용(오프라인 유저에 대해서 적용.)

👉 그래서:

## 8. 롱테일(Long Tail) 개념

### ✔ 정의
> 긴 꼬리(The Long Tail), 또는 롱테일 현상은 파레토 법칙을 그래프에 나타냈을 때 꼬리처럼 긴 부분을 형성하는 80%의 부분을 일컫는다
> 개별은 작지만, 전체 합은 클수도 있다.

---

### ✔ 채팅 시스템에서의 예

* 인기 채팅방 (Head)
* 수많은 소규모 채팅방 (Tail)

👉 대부분 트래픽은 Tail에서도 발생

---
### *키-벨류 저장소의 장점 (채팅 시스템에서 유리한 이유)**
키-벨류 저장소(NoSQL의 한 종류, 예: Redis, DynamoDB, Cassandra)는 데이터를 "키-값" 쌍으로 단순하게 저장해. 채팅 시스템의 롱테일 데이터를 다룰 때 RDB보다 유리한 이유는 다음과 같아:
- **단순성과 유연성**: 키를 메시지 ID(예: "chat:room123:msg456")로, 값을 메시지 내용(JSON 형태로 텍스트, 메타데이터 등)으로 저장해. 스키마가 없어서 다양한 구조의 메시지를 쉽게 저장할 수 있어. 파일에서도 "NoSQL, JSON 컬럼, Key-Value 구조"가 해결 방식으로 언급돼. 롱테일 데이터처럼 구조가 불규칙한 데이터를 유연하게 다룰 수 있어.
- **빠른 성능**: 키 기반 조회가 O(1) 수준으로 빠르고, 쓰기도 단순해. 채팅 시스템에서 오래된 메시지를 조회할 때, RDB처럼 복잡한 쿼리 없이 바로 키로 접근 가능해. 예: "msg:old:2023:01" 같은 키로 오래된 메시지를 그룹화해 저장하면, 롱테일 데이터를 효율적으로 관리할 수 있어.
- **스케일링 용이성**: 수평 확장(샤딩)이 쉬워. 대규모 채팅 시스템에서 트래픽이 증가해도 서버를 추가해 데이터를 분산 저장할 수 있어. 롱테일 데이터가 많아도, 빈도에 따라 스토리지를 최적화(예: SSD for 헤드, HDD for 테일)할 수 있어.
- **비용 효율성**: 롱테일 데이터는 자주 접근되지 않으니까, 저비용 스토리지(예: S3나 Glacier와 통합)와 결합해 사용할 수 있어. RDB처럼 모든 데이터를 고비용 스토리지에 유지할 필요 없어. 채팅 시스템에서는 최근 메시지를 메모리(Redis)나 빠른 스토리지에, 오래된 메시지를 키-벨류에 저장하는 하이브리드 방식으로 비용을 절감해.
- **롱테일 데이터 최적화**: 키-벨류는 데이터의 "빈도"에 따라 저장소를 분리할 수 있어. 예:
  - 헤드(최근 메시지): Redis(메모리 기반)로 빠른 접근.
  - 테일(오래된 메시지): DynamoDB나 Cassandra로 저비용 저장.
  - 이렇게 하면 RDB처럼 모든 데이터를 균등하게 다루지 않아도 돼. 파일의 "Hybrid fan-out 구조"와 유사하게, 데이터 접근 패턴에 맞춰 최적화할 수 있어.
