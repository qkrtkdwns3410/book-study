# 1. 포스팅 전송 (Fanout)

**포스팅 전송(Fanout)**\
→ 어떤 사용자의 새 포스팅을 **그 사용자와 친구 관계에 있는 모든
사용자에게 전달하는 과정**

## Fanout 방식

### 1. Push 모델

특징 - 포스팅이 생성되면 **친구들의 뉴스피드에 미리 push** - 사용자가
피드를 읽을 때 빠르게 보여줄 수 있음

장점 - 빠른 피드 조회 - 읽기 성능이 좋음

단점 - 친구 수가 매우 많은 사용자(celebrity)의 경우 - fanout 비용이 매우
커짐

------------------------------------------------------------------------

### 2. Pull 모델

특징 - 사용자가 피드를 조회할 때 - 친구들의 포스팅을 **그때 모아서
생성**

장점 - 친구가 많은 사용자로 인한 **fanout 과부하 방지**

단점 - 피드 조회 시 계산이 필요함 - 읽기 성능이 상대적으로 느림

------------------------------------------------------------------------

# 2. 포스팅 전송 작업 서버

책에서 표현된 내용 중 의문

> "포스팅 전송 작업 서버"

궁금한 점

-   전송 작업을 해야 하는데
-   도식에서는 **단순 매핑 작업처럼 보임**

가능한 해석

-   실제 시스템에서는 다음 작업을 수행할 수 있음

1.  친구 목록 조회
2.  뉴스피드 대상 사용자 목록 생성
3.  대상 사용자별 뉴스피드 캐시에 post id 저장

즉

**전송 작업 서버 = fanout 작업을 처리하는 worker 서버**

다만 책의 그림에서는 **그 기능 중 일부만 표현된 것으로 보임**

# 3. 그래프 데이터베이스

## 정의

그래프 데이터베이스는

**데이터를 그래프 형태로 저장하는 데이터베이스**

구성 요소

-   Node (엔터티)
-   Edge (관계)
-   Property (속성)

그래프 데이터베이스는

-   엔터티 간 관계를 **명시적으로 저장**
-   수학적 **그래프 이론 기반 연산** 사용

따라서

복잡한 관계가 있는 데이터에서 효율적

예

-   소셜 네트워크
-   사기 탐지
-   추천 시스템
-   지식 그래프

# 5. 관계형 DB vs 그래프 DB 비교

  구분            관계형 데이터베이스           그래프 데이터베이스
  --------------- ----------------------------- --------------------------------
  모델            행과 열 기반                  노드와 엣지
  운영            CRUD + SQL                    CRUD + Graph traversal
  성능            관계 탐색 시 JOIN 비용 증가   관계 탐색 성능 우수
  사용 편의성     정형 데이터에 강함            관계 중심 데이터에 강함
  쿼리            SQL                           Cypher, Gremlin
  스키마          엄격                          비교적 유연
  확장성          수직 확장 중심                관계 데이터에서 수평 확장 유리
  복잡한 관계     JOIN 많아지면 복잡            관계 표현 직관적
  데이터 무결성   ACID 강함                     일부 시스템은 약함
  표준화          SQL 표준                      표준 약함
  생태계          매우 풍부                     상대적으로 제한적
  주요 용도       트랜잭션 처리                 네트워크 분석

# 6. 관계 깊이에 따른 성능 차이

핵심

-   관계 깊이가 깊어질수록 성능 차이가 커짐
-   SQL은 JOIN 증가에 따라 성능이 급격히 저하

  관계 depth   RDB 성능        Graph DB 성능
  ------------ --------------- ---------------
  1 hop        비슷
  2 hop        약간 느려짐
  3+ hop       급격히 느려짐
  4+ hop       Graph DB 압도

이유

-   RDB → JOIN 기반 관계 탐색
-   Graph DB → Edge traversal

# 7. Graph DB 성능 특징

Neo4j 특징

**Index-free adjacency**

즉

각 노드가 **이웃 노드를 직접 가리킴**

그래서

관계 탐색이 거의 **constant time**에 가까움

# 8. Neo4j vs MySQL 성능 실험

사용자 100만명 데이터 기준

  깊이   MySQL 실행시간       Neo4j 실행시간
  ------ -------------------- ----------------
  2      0.016s               0.010s
  3      30.267s              0.168s
  4      1543.505s            1.359s

결과

-   친구의 친구 → Neo4j 약 **60% 빠름**
-   친구의 친구의 친구 → **180배 빠름**
-   depth 4 → **1135배 빠름**
