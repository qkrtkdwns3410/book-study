# 3. 그래프 데이터베이스

## 정의

그래프 데이터베이스는

**데이터를 그래프 형태로 저장하는 데이터베이스**

구성 요소

-   Node (엔터티)
-   Edge (관계)
-   Property (속성)

그래프 데이터베이스는

-   엔터티 간 관계를 **명시적으로 저장**
-   수학적 **그래프 이론 기반 연산** 사용

따라서

복잡한 관계가 있는 데이터에서 효율적

예

-   소셜 네트워크
-   사기 탐지
-   추천 시스템
-   지식 그래프

### 위 그림에 나타난 관계(EDGE)에 대한 해석
아주 간단하게 생각하면 **"문장의 동사(Verb)"**라고 보시면 됩니다.

관계형 DB(RDB)에서는 테이블끼리 그냥 ID 값으로만 엮여 있어서 그 선이 무엇을 의미하는지 한눈에 알기 어렵지만, 그래프 모델에서는 그 선(Edge)에 이름을 붙여서 의미를 부여합니다.

이미지에 나온 3가지 관계의 의미를 문장으로 풀면 다음과 같습니다.

1. OF_PLAYER (선수의...)
* 의미: "이 이적(Transfer) 사건은 **누구(Player)**에 관한 것인가?"를 나타냅니다.
* 해석: (Transfer) -- OF_PLAYER --> (Player)

"이 이적 사건은 '리오넬 메시'라는 선수의 것이다."

* 즉, 이적 노드와 선수 노드를 연결하는 탯줄 같은 역할입니다.

2. FROM_CLUB (~로부터)
* 의미: "이 이적이 시작된 출발지(Origin) 구단은 어디인가?"를 나타냅니다.

* 해석: (Transfer) -- FROM_CLUB --> (Club: FC 바르셀로나)

"이 이적은 'FC 바르셀로나'로부터 시작되었다."

* 화살표의 방향이 이적 사건에서 구단으로 향하며 '출발점'임을 명시합니다.

3. TO_CLUB (~로)
* 의미: "이 이적을 통해 최종적으로 도착한 목적지(Destination) 구단은 어디인가?"를 나타냅니다.

* 해석: (Transfer) -- TO_CLUB --> (Club: PSG)

"이 이적은 'PSG'로 가는 이적이다."

* 화살표가 가리키는 곳이 선수가 새로 둥지를 튼 팀이 됩니다.

# 4. RDB vs Graph DB 구조 비교

| RDBMS | Graph DB |
|-------|----------|
| 테이블 | 레이블(Label) |
| 행(Row) | 노드(Node) |
| 열(Column) | 속성(Property) |
| JOIN | Traversal |

------------------------------------------------------------------------

# 5. 관계형 DB vs 그래프 DB 비교

| 구분 | 관계형 데이터베이스 | 그래프 데이터베이스 |
|------|-------|-------|
| 모델 | 행과 열 기반 | 노드와 엣지 |
| 운영 | CRUD + SQL | CRUD + Graph traversal |
| 성능 | 관계 탐색 시 JOIN 비용 증가 | 관계 탐색 성능 우수 |
| 사용 편의성 | 정형 데이터에 강함 | 관계 중심 데이터에 강함 |
| 쿼리 | SQL | Cypher, Gremlin |
| 스키마 | 엄격 | 비교적 유연 |
| 확장성 | 수직 확장 중심 | 관계 데이터에서 수평 확장 유리 |
| 복잡한 관계 | JOIN 많아지면 복잡 | 관계 표현 직관적 |
| 데이터 무결성 | ACID 강함 | 일부 시스템은 약함 |
| 표준화 | SQL 표준 | 표준 약함 |
| 생태계 | 매우 풍부 | 상대적으로 제한적 |
| 주요 용도 | 트랜잭션 처리 | 네트워크 분석 |

# 6. 관계 깊이에 따른 성능 차이

핵심

-   관계 깊이가 깊어질수록 성능 차이가 커짐
-   SQL은 JOIN 증가에 따라 성능이 급격히 저하

| 관계 depth | RDB 성능 | Graph DB 성능 |
|------------|---------|---------------|
| 1 hop | 비슷 | 비슷 |
| 2 hop | 약간 느려짐 | 약간 느려짐 |
| 3+ hop | 급격히 느려짐 | 빠름 |
| 4+ hop | 매우 느림 | Graph DB 압도 |

이유

-   RDB → JOIN 기반 관계 탐색
-   Graph DB → Edge traversal

# 7. Graph DB 성능 특징

Neo4j 특징

**Index-free adjacency**

즉

각 노드가 **이웃 노드를 직접 가리킴**

그래서

관계 탐색이 거의 **constant time**에 가까움

# 8. Neo4j vs MySQL 성능 실험

사용자 100만명 데이터 기준

| 깊이 | MySQL 실행시간 | Neo4j 실행시간 |
|------|--------|--------|
| 2 | 0.016s | 0.010s |
| 3 | 30.267s | 0.168s |
| 4 | 1543.505s | 1.359s |
| 5 | 1시간 내 완료 실패 | 2.132s |

결과

-   친구의 친구 → Neo4j 약 **60% 빠름**
-   친구의 친구의 친구 → **180배 빠름**
-   depth 4 → **1135배 빠름**

# 9. Graph DB가 적합한 경우

-   관계가 매우 복잡한 데이터
-   multi-hop 탐색이 많은 시스템
