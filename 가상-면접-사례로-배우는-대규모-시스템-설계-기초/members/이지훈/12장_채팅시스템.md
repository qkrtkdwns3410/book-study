# WebSocket 기반 대규모 채팅 서버 설계

###  Http 통신 대신 WebSocket을 쓰는 이유

### HTTP

-   Client → 요청 → Server → 응답 → 연결 종료
-   요청 기반 통신 (request-response)
-   stateless 구조
-   서버가 먼저 메시지를 보낼 수 없음
-   실시간 통신 구현 시 polling / long polling 필요
-   불필요한 요청 반복 발생

### WebSocket

-   Client ↔ Server (연결 유지)
-   연결 기반 통신
-   양방향 통신 (full-duplex)
-   서버 push 가능
-   추가 요청 없이 실시간 데이터 전송 가능


## 핵심 차이

HTTP는 "요청이 있어야 응답 가능"\
WebSocket은 "연결만 되어 있으면 언제든 전송 가능"

→ 채팅은 서버에서 먼저 이벤트가 발생하는 구조이기 때문에 WebSocket이
구조적으로 더 적합


## 문제 정의

WebSocket은 연결을 유지해야 한다.

-   유저 1명 = connection 1개
-   유저 10만명 = connection 10만개
-   유저 100만명 = connection 100만개

단일 서버에서 처리 불가능

## 해결: WebSocket 서버 분산

    Client → Load Balancer → 여러 WebSocket 서버

예시:

    A → 서버1
    B → 서버2
    C → 서버3
    D → 서버1

각 서버는 일부 connection만 관리


## 의문 포인트

같은 서버에 붙어있지 않은데 메시지는 어떻게 전달되는가?

    A → 서버1
    B → 서버2

서버1은 B를 모르고\
서버2는 A를 모름


## 해결 구조 (핵심)

    A → 서버1 → (중앙 브로커) → 서버2 → B

브로커 역할: - Redis Pub/Sub - Kafka

서버 간 메시지 전달 담당

## 같은 서버에 붙어있는 경우

    A → 서버1
    B → 서버1

처리:

    서버1 내부에서 바로 전달

-   브로커 필요 없음
-   네트워크 비용 없음
-   가장 빠름


## 다른 서버에 붙어있는 경우

    A → 서버1
    B → 서버2

처리:

    서버1 → 브로커 → 서버2 → B

-   브로커 필수
-   서버 간 통신 필요


## 메시지 처리 전략

### 전략 1. 모든 메시지 브로커 사용

    A → 서버1 → 브로커 → 모든 서버 → 각 서버가 필터링

특징: - 구조 단순 - 서버 위치 몰라도 됨 - 구현 쉬움 - 불필요한 메시지
전달 발생


### 전략 2. 로컬 확인 후 브로커

    1. 현재 서버에 대상 유저 존재 여부 확인
    2. 있으면 직접 전송
    3. 없으면 브로커 사용

특징: - 네트워크 비용 절감 - 성능 효율적 - 구현 복잡 - 유저 위치 관리
필요

## 클라이언트 위치 문제

클라이언트는 어떤 서버에 붙어있는지 기본적으로 알 수 없음

이유: - Load Balancer가 서버를 선택 - 재연결 시 서버 변경 가능 - 서버
장애 시 이동 발생

## 위치 관리 방식

### 방식 1. 위치 추적 안함

    브로커 → 전체 서버 전파 → 각 서버가 로컬 클라이언트 확인

특징: - 단순 - 안정적 - 추가 관리 필요 없음


### 방식 2. 위치 저장

    userId → serverId

Redis 등에 저장


## 위치 저장 흐름

### 접속 시

    userId → serverId 저장

### 연결 종료 시

    삭제

### 메시지 전송 시

    serverId 조회 → 해당 서버로 전달


## 위치 저장 방식의 문제

-   stale 데이터 발생 가능
-   서버 장애 시 정합성 깨짐
-   재연결 시 업데이트 필요
-   관리 비용 증가
