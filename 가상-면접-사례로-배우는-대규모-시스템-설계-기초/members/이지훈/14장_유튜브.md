# 14장 유튜브 설계 (업로드 & 저장)

## Q. 많은 사용자가 동시에 무거운 영상을 업로드하면 서버는 어떻게 버티냐?

스토리지는 AWS S3를 사용한다고 가정하고,  
우리 서버는 업로드 데이터를 직접 처리하지 않는다.

구조는 다음과 같다.

- 서버는 인증 + 업로드 URL 발급 역할만 수행
- 실제 영상 데이터는 클라이언트 → S3로 직접 업로드

즉, 서버는 데이터 흐름에 끼지 않고 제어 역할만 수행한다.

그래서 트래픽이 아무리 커져도  서버는 “영상 데이터 처리” 때문에 병목이 생기지 않는다.

---

## Q. 그럼 다음 문제

유저는 영상 하나만 올리지만   트랜스코딩을 거치면 여러 개의 파일이 생성된다.

예:
1080p / 720p / 480p / 썸네일 등

---

## Q. 이 파일들은 S3에 어떻게 저장되냐?

S3는 key 기반 저장소다.

그래서 실제로는 다음과 같이 저장된다.

videos/{videoId}/original.mp4  
videos/{videoId}/1080p.mp4  
videos/{videoId}/720p.mp4  
videos/{videoId}/480p.mp4  
videos/{videoId}/thumbnail.jpg  

이 구조는 계층 구조가 아닌, 실제로는 단순한 문자열 key일 뿐이다.

---

## Q. 그럼 우리는 이걸 어떻게 관리하냐?

핵심은 다음과 같다.

- S3는 저장만 담당
- 구조 관리는 DB에서 담당

예시: 유저가 영상을 업로드할 경우

{
  "videoId": 123,
  "original": "videos/123/original.mp4",
  "resolutions": [
    "videos/123/1080p.mp4",
    "videos/123/720p.mp4",
    "videos/123/480p.mp4"
  ]
}

즉, 유저와 영상등의 관계는 DB 에서 관리하고, S3에는 KEY하나만으로 PUT을 한다고 한다.

KEY 하나만으로, 엄청 많은 영상을 커버할수 있다고?
: 이부분은 S3가 내부적으로 샤딩 + 파티셔닝을 한다고 한다.

### 1. DB 테이블 설계

영상 시스템 최소 구성

- User
- Video
- VideoFile

---

#### user

user
- user_id (PK)
- email
- created_at

---

#### video

video
- video_id (PK)
- user_id (FK)
- title
- status (UPLOADING / PROCESSING / READY / FAILED)
- created_at

---

#### video_file

video_file
- id (PK)
- video_id (FK)
- quality (original / 1080p / 720p / 480p / thumbnail)
- s3_key
- file_size
- created_at


### 2. 업로드 이후 흐름

#### 1) 영상 생성 요청

Client → Server  
POST /videos

- video_id 생성
- video.status = UPLOADING 저장
- S3 pre-signed URL 발급

---

#### 2) S3 업로드

Client → S3 (PUT)

videos/{videoId}/original.mp4

---

#### 3) 업로드 완료 처리

Client → Server  
POST /videos/{videoId}/complete

- video.status = PROCESSING
- video_file(original) 생성

video_file
- video_id = 123
- quality = original
- s3_key = videos/123/original.mp4

---

#### 4) 트랜스코딩 (비동기)

S3 → Queue → Transcoder

결과 파일 생성:

videos/123/1080p.mp4  
videos/123/720p.mp4  
videos/123/480p.mp4  

---

#### 5) 트랜스코딩 결과 저장

video_file 테이블에 추가

(video_id=123, quality=1080p, s3_key=...)  
(video_id=123, quality=720p, s3_key=...)  
(video_id=123, quality=480p, s3_key=...)  

---

#### 6) 최종 상태 변경

video.status = READY


---

## 하나의 영상당 해상도별 * K 의 영상이 생기는데 너무 비효율적이지 않은가

영상 하나 올렸는데  
파일이 4~5개로 늘어나면 저장 용량도 그만큼 증가한다. (실제로 용량은 늘어난다.)

하지만 이과정이 필요한 이유

### 1. 스트리밍 최적화
네트워크 환경에 따라 다른 화질을 제공해야 한다.  
이걸 안하면 영상 끊김이 발생한다.

### 2. 사용자 경험
끊기거나 느린 영상은 바로 이탈로 이어진다.  
저장 비용보다 사용자 경험이 더 중요하다.

### 3. CDN 효율
자주 보는 해상도만 캐싱된다.  
모든 파일이 동일하게 사용되는 것은 아니다.

### 4. 선택적 트랜스코딩
모든 해상도를 항상 생성하지 않는다.  
영상 길이, 품질, 조회수 기준으로 일부만 생성하기도 한다.
