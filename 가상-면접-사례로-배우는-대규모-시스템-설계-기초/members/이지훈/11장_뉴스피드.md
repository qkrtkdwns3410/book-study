# Fanout-on-write vs Fanout-on-read  
## Instagram Feed 시스템 설계 (DB 설계보단 트래픽 관점에서)

## 1. 문제 정의

팔로워가 수십만~수천만명인 유명인이 게시글을 작성할 경우

- 게시글 저장
- 피드 생성
- 사용자 조회 처리


SNS 피드는 다음 구조를 가진다.

- 한 사용자의 게시글 작성 → 수많은 사용자에게 전달
- Read 트래픽이 Write보다 훨씬 많음

즉

```
1 Write → Millions Read
```

이 구조 때문에 Fanout 전략이 필요하다.

---

## 3. Fanout-on-write (Push 모델)

### 개념

게시글 생성 시점에 미리 피드를 생성한다.

### 게시글 작성 Flow

1. 게시글 DB 저장
2. 작성자의 follower 목록 조회
3. 각 follower의 timeline cache에 게시글 push
4. 피드 조회 시 cache만 읽음

### 특징

- Read latency 매우 낮음
- 즉시 피드 반영 가능

### 문제

팔로워가 많으면 write 비용이 폭발한다.

예:

```
Follower 1,000,000명
→ 게시글 1개 작성 시
→ 1,000,000번 feed write 발생
```

---

## 4. Fanout-on-read (Pull 모델)

### 개념

피드를 조회할 때 생성한다.

### 게시글 작성 Flow

1. 게시글 DB 저장
2. 추가 작업 없음 (fanout skip)

### 피드 조회 Flow

1. 사용자 following 목록 조회
2. 각 팔로워의 최신 게시글 조회
3. Merge + Sort 
4. 응답 반환

### 특징

- Write 비용 매우 낮음
- 쓸떄 팔로워 수만큼 캐시 ID를 만드니까 ID duplicate 감소


### 문제

- Read latency 증가
- Query 비용 증가
- Merge 비용 발생

---

## 5. Celebrity 문제

SNS에서는 대부분 사용자가 follower 수가 적지만  
일부 유명인은 follower가 수백만~수천만에 달한다.

이때 fanout-on-write를 그대로 적용하면

- message queue 폭발 (유명인이 글쓰면 저자앟는데 한세월..)
- hot partition 문제 발생 (하나의 ID에 엄청난 Cache write 진행)

따라서 유명인은 별도 전략이 필요하다.

---

## 6. Hybrid Fanout 전략

실제 대규모 SNS는 다음 전략을 사용한다.

```
일반 사용자 → Fanout-on-write
유명인 → Fanout-on-read
```

### Celebrity 판단 기준 예시

- follower_count > 1,000,000 (훨씬 좋은 threashold 알고리즘이 있겠지만)

---

## 7. Celebrity 게시글 작성 Flow (Hybrid)

### POST /post 요청

1. 게시글 DB 저장
2. user.isHighFanoutAccount 여부 확인
3. celebrity인 경우 fanout 작업 skip
4. 일반 사용자일 경우 fanout queue publish (일반 사용자는 팔로워가 적으니, 팔로워 timeline cache에 저장해도 부하가 적다.)

즉

```
Celebrity → write fanout 없음
Normal User → write fanout 수행
```

---

## 8. Feed 조회 Flow (Celebrity 포함)

사용자가 앱을 열면

### GET /feed 요청 처리

1. timeline cache 조회 (일반 사용자 게시글)
2. following 중 celebrity 목록 조회
3. celebrity 최신 게시글 cache / DB 조회
4. Merge + Ranking 수행
5. 응답 반환

즉

```
부분 precomputed feed (일반 팔로워) + on-demand celebrity merge (유명인 팔로워)
```

구조가 된다.

---

## 9. 시스템 아키텍처 예시

```
Client
  ↓
API Gateway
  ↓
Feed Service
  ├── Timeline Cache (Redis)
  ├── Celebrity Post Cache
  └── DB (RDBMS)

Write Path
User → Post Service → DB
                 → (if normal) Queue → Feed Fanout Worker → Timeline Cache

Read Path
User → Feed Service → Cache-DB Merge → Response
```
