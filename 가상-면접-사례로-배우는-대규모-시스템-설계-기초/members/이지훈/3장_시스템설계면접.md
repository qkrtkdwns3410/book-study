시스템 설계 면접 진행 흐름 요약

1. 문제 이해 및 설계 범위 확정

요구사항을 정확히 이해하지 않은 상태에서 바로 설계로 들어가지 않는다.

* 추가로 확인할 질문들

  * 핵심 기능 범위 (정렬 기준, 최대 처리량, 미디어 포함 여부 등)
  * 사용자 규모 (MAU, DAU)
  * 회사 규모 및 서비스 단계
  * 사용 중인 기술 스택
  * 기존에 활용 가능한 인프라나 서비스 존재 여부

이 단계의 목적은 설계 범위를 명확히 고정하는 것이다.

2. 개략적 설계안 제시 및 합의

면접관은 평가 대상이 아니라 협업 상대다.

* 전체 구조의 첫 청사진 제시
* 핵심 컴포넌트 포함

  * 클라이언트
  * API / 웹 서버
  * 데이터 저장소
  * 캐시
  * CDN
  * 메시지 큐 등
* 제약 조건을 만족하는지 확인

  * 필요한 경우, 사전 계산이 필요한지 먼저 질문
* 규모에 따라

  * API 엔드포인트 수준까지만 논의할지
  * DB 스키마까지 들어갈지 결정

예를 들어 뉴스피드라면

* 피드 생성
* 피드 발행
  두 단계로 흐름을 나누는 수준까지면 충분하다.

3. 상세 설계

불필요한 디테일에 시간을 쓰지 않는다.

* 핵심 컴포넌트부터 우선순위 지정
* 개략적 구조에 필요한 수준의 디테일만 추가
* 병목 가능 지점 위주로 설명

4. 마무리 및 확장 논의

완벽한 설계는 없다.

* 전체 설계 흐름 간단히 요약
* 장애 발생 시 영향 범위 점검
* 운영 관점 논의

  * 메트릭 수집
  * 모니터링
  * 로그
  * 배포 방식
* 향후 트래픽 증가 시 확장 방향 제시

이 단계에서는 실무 감각을 보여주는 게 핵심이다.
